# NPC 简化流程设计

**日期**：2026-06-26
**状态**：待实现
**目标**：先用最小机制验证 NPC 对话效果，去掉内心独白节点和复杂人物设定，降低调试变量。

---

## 背景

当前 NPC 对话使用 LangGraph 双节点流程：

1. `think_node` 调用模型生成内心独白。
2. `respond_node` 再基于内心独白生成玩家可见回复。

这套机制适合后续做复杂行为控制，但现在会增加效果判断难度：一次玩家输入会触发两次模型调用，且 `/review` 会暴露内心独白，容易把调试视角和玩家体验混在一起。

本轮目标是先做简化版，让你快速测试“只靠人物基础设定 + 对话历史 + 回复 prompt”的实际效果。

---

## 范围

### 本次包含

- 去掉运行时内心独白节点。
- 对话流程从 `think -> respond` 简化为单次回复。
- 人物设定生成改为最小字段。
- `/review` 模式不再展示内心独白，只展示玩家问题和 NPC 回复。
- 在分支中完成和测试，确认可用后再合入主干。

### 本次不包含

- 新增前端或 UI。
- 新增模型供应商。
- 引入新的 LangGraph 节点。
- 做复杂防御策略、封闭话题、心理推理、威胁等级等机制。
- 删除测试体系或游戏主流程。

## 架构设计

### 对话图

推荐把运行时图简化为单节点：

```text
player_input -> respond -> END
```

`think_node` 不再参与运行。这样每轮只调用一次模型，效果更直接，调试也更容易。

### 兼容策略

为了减少一次性改动范围，`run_npc_turn()` 可以暂时保持原返回结构：

```python
tuple[str, str]  # (inner_monologue, npc_response)
```

其中 `inner_monologue` 固定返回空字符串。这样 `GameSession` 等调用方不需要大范围重写，但对外行为已经不再依赖内心独白。

后续如果确认简化路线稳定，再单独做接口清理，把 `inner_monologue` 从数据结构中彻底移除。

---

## 人物设定

人物 profile 先保留最少字段：

- `name`
- `age`
- `occupation`
- `city`
- `personality`
- `speech_style`

移除或暂时不用复杂字段：

- `backstory`
- `hobbies`
- `defense_style`
- `sealed_topics`
- `quirks`

生成 prompt 也同步简化，只要求模型输出基础身份、性格和说话风格。目标不是一次生成“完整人物传记”，而是先测试对话回复是否自然。

---

## Prompt 设计

### 回复 prompt

`respond_node` 直接接收：

- 简化后的 profile system prompt
- 对话历史
- 当前玩家输入

回复要求保持简短：

- 不提自己是 AI 或机器人。
- 不输出动作描写或括号舞台提示。
- 不主动堆背景细节。
- 用人物的说话风格自然回答。
- 对怀疑或试探类问题可以简短带过、反问或转移，但不需要显式威胁等级。

### 删除内心独白依赖

`build_respond_prompt()` 不再需要 `inner_monologue` 参数参与 prompt 内容。为了兼容旧调用，可以先保留参数但忽略它，或者同步更新调用签名。实现计划阶段再根据测试影响选择更小改动路径。

---

## Review 模式

`/review` 保留为对话回看模式，但去掉内心独白展示。

每轮只展示：

- 玩家问题
- NPC 回复

不再显示：

- 内心独白面板
- 与 think 节点相关的标题或说明

如果 `TurnRecord` 仍暂时保留 `inner_monologue` 字段，该字段固定为空，并且 review UI 不读取它。
